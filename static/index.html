zimapp

v2

Throw in a URL to a docker-compose.yml (or a Dockerfile) — zimapp turns it into a ZimaOS-conformant compose with an x-casaos block, checks it against the eight rules and installs it. Multi-service stacks (app + database) stay one app: x-casaos.main points at the WebUI service.

1 · Source

GitHub/GitLab blob links are flipped to their raw variant automatically. Open .yml reads a file from this computer — a typed path would be looked for inside the zimapp container, not on your machine.

2 · Detected stack

main serves the WebUI and gets port_map · support is infrastructure (database, cache, queue).

3 · App metadata

Ends up in the x-casaos block — that is what ZimaOS shows on the tile.

4 · Variables from the source

ZimaOS ships no .env with it — every unresolved ${VAR} is a startup failure, not a cosmetic flaw. Empty secret fields are filled with a random value on request and shown below in plain text.

5 · Target system

Uses the app grid API (credentials from step 7 — works without SSH, but sees only ZimaOS apps) and docker via SSH (needs a key login, sees foreign containers too). Whichever answers is used; the result says which one did.

7 · Install on ZimaOS

POST /v2/app_management/compose with raw YAML. The installation runs asynchronously — HTTP 200 means “accepted”, not “done”. Credentials are used for this request only and are stored nowhere.

Already installed? Compare with installation reads the compose ZimaOS really has and shows what would change. Apply update then changes it in place (PUT) — images and /DATA/AppData/<app>/ survive, which an uninstall+install does not.
